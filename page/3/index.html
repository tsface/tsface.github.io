Structured Streaming Kafka Source 源码分析

Posted on 2018-01-05 Edited on 2019-12-06

KafkaSourceProvider

自定义SourceProvider可以向Structured Streaming注册自定义Source，其实现了StreamSourceProvider和 DataSourceRegister接口。需要重写以下三个方法：

  • sourceSchema
    获取数据源的schema信息，返回类型是Tuple[_,_],其中的key代表数据源的别名，value表示数据源的schema
  • createSource
    创建一个org.apache.spark.sql.execution.streaming.Source接口的实现，表示数据源对象
  • shortName
    自定义的数据源名称，这个数据源名称会提供给用户的format方法设置数据源

LogManager分析

LogManager是kafka的日志管理子系统，此系统负责日志的创建，恢复和清理。所有的数据读写操作被委派给具体的日志实例。日志管理子系统维护一个或者多个日志目录，新文件在数据最少的日志目录中创建。
其内部有一个后台线程负责周期性的处理日志回滚操作。

Spark Streamig backpressure 机制

Posted on 2016-12-24 Edited on 2019-12-06

Spark Streamig backpressure 机制

一、背景

Spark Streaming实现流式处理的最基本原理就是周期性，持续的生成任务来处理数据。在这个过程有两个概念：任务生成周期(batch interval)和数据处理时间(batch processing time)。如果数据处理时间大于任务提交间隔,意味着数据的接收速率已经大于数据的处理速度了，如果数据的存储策略是内存(MEMORY_ONLY)，则会导致内存数据丢失，数据丢失；如果数据存储策略是内存和磁盘，则数据会溢写到磁盘，降低数据处理性能。
在1.5.x版本之前采取的是最简单的方式：即限流(spark.streaming.receiver.maxRate)，通过这个配置项减小数据接收的速率，防止数据处理太慢，数据接收过快而导致的一系列问题。但是这种方式也有自己的局限，当生产环境中数据生产速率高于maxRate，当集群处理能力高于maxRate，这就会导致集群资源浪费，使用不合理等。所以后续产生了动态控制数据接收速率的机制-backpressure
